Для фиксации изменений в системе и ведения историй изменений необходимо вести жрнал логгирования основных систем. В проекте "Александрит" в первую очередь надо добавить логгирование в сервисе **MES API** и **CRM API**. Это ключевые узлы системы, которые отвечают за бизнес-логику и необходима фиксация всех изменений в процессах. Все сообщения должны содержать поле "источник сообщения" - это программный модуль, метод или логическая сущность, ответственный за формирование сообщения.

Логгирование **MES API**:

уровень INFO:

- посчет стоимости заказа (PRICE_CALCULATED). Логгируем время, источник сообщения, идентификатор заказа, рассчетную стоимость;

- оператор взял заказ в работу (MANUFACTURING_STARTED). Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора, детали заказа;

- оператор выполнил заказ (MANUFACTURING_COMPLETED). Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора, детали заказа;

- оператор начал упаковывать заказ (PACKAGING). Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора, детали заказа;

- заказ отправлен покупателю (SHIPPED). Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора, трек номер доставки (логгировать данные доставки нельзя. это конфиденциальная информация);

уровень WARNING:

- оператор отказался от заказа. Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора, детали заказа;

- вызовы API, которые вернули статус логической ошибки. Логгируем время, источник сообщения, TraceId вызова если есть, текст ошибки.

уровень ERROR:

- статус заказа сменился на ошибочный. Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора если есть, причина смены статуса;

- критические ошибки в вызовах API. Логгируем время, источник сообщения, TraceId вызова если есть, текст ошибки;

уровень FATAL:

- выход из строя приложения.

Логгирование **CRM API**:

уровень INFO:

- регистрация заказа в системе. Логгируем время, источник сообщения, идентификатор заказа, параметры заказа.

- заказ подтверждён (MANUFACTURING_APPROVED). Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора;

- заказ помещен в очередь RabbitMQ. Логгируем время, источник сообщения, идентификатор заказа;

- заказ получен из очереди RabbitMQ. Логгируем время, источник сообщения, идентификатор заказа;

- заказ завершен (CLOSED). Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора, трек номер доставки (логгировать данные доставки нельзя. это конфиденциальная информация);

уровень WARNING:

- заказ отменен оператором. Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора,  причину отмены если есть;
- заказ возвращен получателем. Логгируем время, источник сообщения, идентификатор заказа, идентификатор оператора, трек номер доставки, причину отказа если есть (логгировать данные доставки нельзя. это конфиденциальная информация);
- вызовы API, которые вернули статус логической ошибки. Логгируем время, источник сообщения, TraceId вызова если есть, текст ошибки.

уровень ERROR:

- критические ошибки в вызовах API. Логгируем время, источник сообщения, TraceId вызова если есть, текст ошибки;

уровень FATAL:

- выход из строя приложения.

# **Мотивация**

Внедрение логгирования позволит нам быть в курсе изменений, происходящих в системе. Это даст нам следующие инструменты и кейсы.

- Разбор изменений систем в моменты критических сбоев. В случае массовых инцидентов, наличие логов позволит быстрее выявить источник проблем. Для этого в журнале сообщений нужно сделать фильтр по уровню ERROR и FATAL. Источник сообщений и будет причиной сбоя. 

- Выбирая фильтр о источнику сообщений, можно проанализировать производительность, надежность и корректность работы программных модулей. Для этого можно использовать логи уровня DEBUG на тестирование и при проведении нагрузочного тестирования.

- Фильтр по идентификатору заказа, может предоставить общую картину жизненного цикла заказа, что может быть полезно при тестировании или при расследовании по обращению пользователей.

- Фильтр по идентификатору оператора позволит проанализировать качестро работы операторов, их производительность и их "недобросовесность".

# **Предлагаемое решение**

Логгирование огранизуется на стеке OpenSearch. 

1. Разработчики реализуют с помощью специальных библиоте запись логов в формате json в файлы. Максимальный размер файлов должен быть ограничен.

2. Устанавливается и настраивается Filebeat для сбора файлов логов приложений. Собранные логи отправляются им на Logstach

3. Logstach собирает и сохраняет сформированные логи в базу OpenSearch. 

4. В OpenSearch задаются индексы, для оптимизации коллекции.

5. OpenSearch Dashboards визуализирует собранные логи для службы поддержки и разработчиков.

### Безопасность

Данные логов должны быть доступны пользователям из группы "Разработчики" и "Руководители". 

Данные в логах не должны содержать персональные данные (ФИО пользователей, адреса, номера счетов). Все сущности сохраняются только с использованием их индентификаторов (id) в системе. 

Данные в логах не должны содержать секретные данные и настройки системы (пароли и логины доступов к бд, к API, к сеансам).

### Политика хранения логов

В системе Elasticsearch предлагается настроить индексы для каждой системы. Разработчки системы будут иметь быстрый доступ к логам своей системы.

Время хранения логов на продакшене должно перекрывать время жизни заказа в 2 раза. Например, если 95% заказов создаются и завершаются в течении 1 месяца, то время хранениея логов должно быть минимум 2 месяца.

Время хранения логов на тестовой площадке 2 недели. Это в двое превышает время тестирования релиза. 

### Алертинг

Нужно настроить алертинг для разработчиков на превышение числа ошибок в системе.

Нужно настроить алертинг для разработчиков для событий уровня FATAL.
