# **Мотивация**

Система мониторинга позволяет контролировать состояние системы. Предлагается контролировать следующие параметры:

- состояние железа (параметры работы серверов приложений и баз данных):
  
  - загрузка процессора;
  
  - состояние памяти;
  
  - загрузка оперативной памяти.

- состояние элементов системы :
  
  - загрузка баз данных;
  
  - параметры работы баз данных;
  
  - хелсчеки сервисов;

- сетевая активность и мониторинг запросов: 
  
  - общая сетевая активность на сервисах;
  
  - успешные запросы;
  
  - ошибочные запросы;
    
    - время откликов запросов.

Настройка мониторинга позволит решить следующие проблемы:

- мониторинг позволит выявить узкие места в системе. Изучив их мы сможем оптимизировать их, тем самым повысив скорость работы и надежность системы.

- анализ производительности подскажет где надо наращивать систему для дальнейшего роста. Это позволит приобрести только необходимое оборудование для роста бизнеса.

- анализ активности пользователей позволит бизнесу уточнять бизнес-метрики;

# **Выбор подхода к мониторингу**

Выделим основные виды мониторинга.

1. Мониторинг баз данных. Очень важный параметр, от которого зависит быстродействие и надежность все системы. Сбои в работе баз данных могут быть череваты потерями денег и репутации.

2. Мониторинг производительности. Этот параметр позволяет выявлять узкие места в работе сервисов и предоставляет данные для анализа их работы, сбоях и нагруженных местах. 
   
   1. Анализ производительности **MES** сервиса позволит оптимизировать его работу и повысить скорость формирования заказов. Тут лучше использовать **USE** метод мониторинга. Мониторим средннее время и объем работы сервиса. Это позволит проанализировать работу сервиса на разных заказах. Контролируя время работы сервиса по каждому заказу, мы сможем проанализировать и узнать причины долгой работы.
   
   2. Анализ производительности **CRM** сервиса подойдет **Четыре золотых сигнала**  для анализа запросов пользователей и других систем. Это покажет трафик, который проходит через сервис и даст жанные для масштафирования. Качество работы сервиса, мы получим через число ошибок, задержки и насыщенность сервиса. 
   
   3. Анализ производительности **Онлайн-магазина** сервиса подойдет **Четыре золотых сигнала** для анализа запросов пользователей и других систем. Анализ трафика покажет как масштабировать сервис. Какие нагрузки надо выдерживать и какой запас производительности надо иметь.

3. Сетевой мониторинг позволит просмотреть общее состояние системы и показать наличие сбоев в ее работе. Найти сервисы, требующие масштабирования или доработки.   

Контролируемые параметры:

1. CPU % for shop API - высокая загрузка процессора покажет проблемы с кодом или возросшую нагрузку;
2. CPU % for CRM API - высокая загрузка процессора покажет проблемы с кодом или возросшую нагрузку;
3. CPU % for MES API - высокая загрузка процессора покажет проблемы с кодом или возросшую нагрузку;
4. Size of S3 storage - контроль размера хранилища подскажет, когда пора расширяться;
5. Size of shop db instance - контроль размера бд подскажет, когда надо оптимизировать хранение данных в БД;
6. Size of MES db instance - контроль размера бд подскажет, когда надо оптимизировать хранение данных в БД;
7. Number of connections for shop db instance - число подключений в БД важный параметр, при превышении которого начнутся проблемы с сохранностью данных и получения;
8. Number of connections for MES db instance - число подключений в БД важный параметр, при превышении которого начнутся проблемы с сохранностью данных и получения;
9. Number of `HTTP 200` for shop API - число успешных запросов показывает нагрузку, которую выдерживает сервис;
10. Number of `HTTP 200` for CRM API - число успешных запросов показывает нагрузку, которую выдерживает сервис;
11. Number of `HTTP 200` for MES API - число успешных запросов показывает нагрузку, которую выдерживает сервис;
12. Number of `HTTP 500` for shop API - число ошибочных запросов покажет проблемы в работе сервиса. Требуется установить алертинг на превышение данного параметра;
13. Number of `HTTP 500` for CRM API - число ошибочных запросов покажет проблемы в работе сервиса. Требуется установить алертинг на превышение данного параметра;
14. Number of `HTTP 500` for MES API - число ошибочных запросов покажет проблемы в работе сервиса. Требуется установить алертинг на превышение данного параметра;
15. Number of `dead-letter-exchange` letters in RabbitMQ - число неотправленных сообщений покажет проблемы с переполнением очереди сообщений;
16. Number of requests (RPS) for internet shop API - число запросов в секунду покажет загруженность сервиса и позволит принять решение о масштабировании данного узла;
17. Number of requests (RPS) for CRM API - число запросов в секунду покажет загруженность сервиса и позволит принять решение о масштабировании данного узла;
18. Number of requests (RPS) for MES API - число запросов в секунду покажет загруженность сервиса и позволит принять решение о масштабировании данного узла;
19. Number of requests (RPS) per user for internet shop API - число запросов от пользователя позволит противодействовать DDos атакам или предупредить о некорректной работе клиента;
20. Number of requests (RPS) per user for CRM API - число запросов от пользователя позволит противодействовать DDos атакам или предупредить о некорректной работе клиента;
21. Number of requests (RPS) per user for MES API - число запросов от пользователя позволит противодействовать DDos атакам или предупредить о некорректной работе клиента;
22. Response time (latency) for shop API - время отклика сервиса покажет проблемы с работоспособностью определенных частей API;
23. Response time (latency) for CRM API - время отклика сервиса покажет проблемы с работоспособностью определенных частей API;
24. Response time (latency) for MES API - время отклика сервиса покажет проблемы с работоспособностью определенных частей API;
