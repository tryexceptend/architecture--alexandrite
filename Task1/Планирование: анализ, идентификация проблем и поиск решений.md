# Анализ

В схеме архитектуры компании «Александрит»можно выделить следующие проблемные места.

1. **Нет мониторинга.** Поскольку система испытывает трудности, надо выяснить узкие места и проблемные зоны. Без хорошего мониторинга системы это сделать будет проблематично.

2. **Нет логгирования и трейсинга**. Логгирование и трейсинг запросов помогли бы контролировать систему, помогали быстро идентифицировать проблемы в системе. Трейсинг позволит определить где "теряются" заказы и происходят сбои. 

3. **Нет репликаций БД (баз данных)**. Все базы имеют по одному инстансу, которые работают на чтение и запись одновременно. Это может быть узким местом, при возрастании нагрузки и объемов данных.

4. **Нет возможности горизонтального масштабирования приложений**. Все экземпляры приложений запущены в единичных экземплярах на ВМ (виртуальная машина). При растущих нагрузках мы будем упираться в ограничения ВМ.

5. **Ручное тестирование**. Один QA-инженер перестает справляться с тестированием продукта. 

6. **Ручной деплой в продакшен**. Развертывание системы происходит вручную, силами одного DevOps-инженера. Это заставляет деплоиться по ночам и останавливать работу системы. Ручное развертывание черевато ошибками.

# Инциативы

На основе указанных выше проблемных мест предлагаются следующие инциативы.

1. **Настроить мониторинг.** Необходимо настроить мониторинг всех элементов системы. 
   
   - Для мониторинга API сервисов подойдет стратегия "Четыре золотых сигнала". Мы мониторим трафик, ошибки и задержки в обработке запросов. Это покажет как работают сервисы, их стабильность и подсветит узкие и проблемные места;
   
   - Для мониторинга БД стратегия USE. **Utilization** - мониторим число подключений, число активных транзакций, объем трафика. С помощью мониторинга можно определить какие стратегии оптимизации БД подойдут (шардирование, репликация, кеширование). Подсветит неоптимизированные запросы от сервисов. Мониторинг числа подключений и объема данных позволит своевременно принять решение о  масштабировании и определиться с характеристиками требуемого оборудования. Число открытых, но не используемых соединений может подсветить некорректное использование ресурсов сервисами. **Saturation** (насыщенность) - число ожидающих подключений, число ожидающих транзакций. Покажет проблемы с числом подключений к БД и ошибки в сервисах. **Errors** (ошибки) - число неуспешных соединений, число ошибочных запросов, число ошибочных транзакций. Это может подсветить проблемы в логике, проблемы с использованием ресурсов БД в приложении. Так же нужен контроль используемой памяти, оперативки и CPU. Это позволит прогнозировать дальнейший рост и масштабирование БД.
   
   - Мониторинг фронтендов по стратегии RED позволит выявить проблемы в пользовательских приложения, покажет недобросовесных пользователей. Это позволит сделать систему более отзывчивой.  

2. **Реализовать репликацию баз данных.** Необходимо реализовать репликацию существующих БД. Это позволит решить несколько проблем и предоставит следующие преимущества:
   
   - повысит надежность системы. Данные не пропадут в случае выхода из строя оборудования.
   
   - позволит разнести операции записи и чтения. Запись будет проводиться на мастер, а чтение данных со слейва. Это повысит отклик системы и повысить SLI.

3. **Найм/обучение DevOps-инженера и Тестировщика.** С ростом проекта будет увеличиваться размеры и число БД, появятся новые системы и оборудование, которое потребуется мониторить и обслуживать. Силами одно человека сделать это будет проблематично. В рамках найма будет следующего опыта у кандидата:
   
   - опыт работы с БД PostgreSQL;
   
   - опыт работы с S3 хранилищами;
   
   - опыт настройки API Gateway (Nginx, Kong, Consul);

4. **Автоматическое тестирование.** Для ускорения релизов необходимо автоматизировать процессы сборки, тестирования и развертывания для всех типов окружения (CI). У тестировщика уже есть наработки в эту сторону. Надо помочь ему и нанять в помощь ему еще одного тестировщика.

5. **Автоматический деплой.** Настройка автоматизированного деплой на продакшен позволит избежать ошибок при развертывании приложений. Это повысит стабильность системы и деплой будет происходить без остановки работы.

6. **Внедрение кеширования.** Кеширование высоконагруженных элементов системы позволит ускорить их работоспособность, повысить отзывчивость и стабильность. Но это потребует значительных затрат на разработку и тщательное тестирование.

7. **Доработка ПО для возможности горизонтального масштабирования.** Команды разработки ПО и DevOps-инженеры должны рассмотреть возможность реализации компонентов системы с возможностью запускать несколько инстансов приложений. Возможно для этого потребуется только использование API Gateway, который будет распределять нагрузку между несколькими инстансами приложения. Возможно потребется доработка самих приложений.

Первые три пункта для реализации я вижу такие:

1. Внедрение мониторинга - это поможет в решении текущих проблем и облегчит жизнь в будующем;

2. Оптимизация баз данных - обычно БД это узкое место в системе. Их корректная работа и оптимальное использование положительно скажется на работоспособности системы;

3. CI/CD - автоматизация процессов сборки, тестирования и развёртывания ПО, позволит быстро выпускать новые функции и исправления, исключит ошибки связанные с человеческим фактором, защитит систему.

В итоге получится система которая будет справляться с нагрузками и масштабируемая. В ней будет:

- Мониторинг всех систем;

- Логгирование работы сервисов;

- MES сервис с кешем, для быстрой работы;

- Сервисы можно масштабировать горизонтально;

- Все БД реплицированы и работают по схеме: мастер на запись, слейвы на чтение;

- Все процессы тестирования и развертывания будут происходить автоматически (CI/CD), возможно даже в контейнерах
