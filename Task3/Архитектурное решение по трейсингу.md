Используем для внедрения трейсинга платформу **OpenTelemetry (OTel)**. 

Трейсин следует добавить с следующие сервисы системы:

- Internet Shop - запросы формирования и редактирования заказа;

- Shop API - запросы формирования и редактирования заказа

- CRM API - запросы подтверждения заказа, отправка заказа в производство, завершение заказа;

- MES - запросы от пользователей к MES API по изменению заказа;

- MES API - запросы по изменению заказа.

Возможные места отказа это API сервисы, которые перегружены или содержат ошибки. Следует внедрить трейсинг на запросы этих сервисов, которые влияют на работу с заказами.

Контекст трейсинга должен содержать следующие данные:

- в качестве trace_id - идентификатор заказа;

- статус заказа;

- ендпоинт API метода;

- если есть идентификатор оператора;

- если есть идентификатор доставки;

- статические данные пользователей (ip);

- ошибочные ответы от API.

# **Мотивация**

Внедрение трейсинга запросов позволит выявить проблемы в пути, который проходит заказ в системе. С его помощью мы увидим где больше всего задерживаются запросы, какие возникают ошибки и сбои в работе сервисов.

Поддержка сможет быстрее выявить место, где "оборвался" или "завис" заказ. Зайдя в Jaeger специалист поддержки по идентификатору заказа быстро поймет в каком статусе заказ, где он сейчас обрабатывается и какая ошибка в его обработке могла быть.

Мониторинг задержек в обработке запросов позволит разработчикам определить проблемные места в API. Они могут проработать решения по их усовершенствованию или масштабированию сервиса.

Трейсинг ошибок позволит выявить недобросовесных клиентов или операторов, которые пытаются воспользоваться дырами в системе. Расследование и анализ трейсов помогут выявить их.

Анализ запросов поможет оптимизировать некоторые узлы системы, если запросы к ней слишком частые и содержат однотипные данные. В таких местах возможно будет удобно внедрение кеша, что снизит нагрузку на ядро системы и ускорит отклик у конечного пользователя.  

# **Предлагаемое решение**

Предлагается внедрить трейсинг запроов с помощью платформы **OpenTelemetry (OTel)** и **Jaeger**. 

Разработчики добавляют SDK OTeL в системы и настраивают наполнение контекста трейсов запросов, связанных с заказами. 

В качестве коллектора используем Jaeger с хранилищем данных в ClickHouse. Для визуализации метрик используем Jaeger-UI. Используем функционал доступный "из коробки".

# **Компромиссы**

Это решение потребует затрат:

- на доработку системы по трейсинг;

- на новое хранилище данных;

- на docker-контейнеры с Jaeger.

Это даст хороший инструмент для разработчиков и DevOps-ов для мониторинга состояния и работоспособности системы. Позволит выявлять узкие места и планировать оптимизацию. Но для поддержки потребуется проводить обучение Jaeger-UI. Если для них или для бизнеса потребуется доработка, то это потребует значительных затрат, поскольку это затронет все узлы системы.

# Безопасность

Данный функционал должен быть доступен по авторизации для групп "Разработчики" и "Поддержка". Это основные бенефициары данной системы.
